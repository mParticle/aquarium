# UX Patterns

UX Patterns are **composed recipes** built from the primitives in the Components section. They solve recurring product problems — filtering data, displaying stats, running a multi-step flow, restricting access — by stitching several components together into a proven layout with agreed-on behavior.

Think of them this way:

- **Components** are the vocabulary (Button, Input, Card, Tooltip).
- **UX Patterns** are the sentences (Date Range Filter, Statistics Card, AI Copilot Empty State).

Patterns are opinionated by design. If a team keeps reaching for the same combination of primitives, we promote that combination here so every product surface behaves the same way.

## Groups

- **Data Exploration** — filtering, selecting, and surfacing metrics.
- **Flows** — multi-step experiences, landing pages, and AI-driven conversations.
- **Inline Controls & States** — small, reusable affordances for help, permissions, and row-level actions.

## When to reach for a pattern

- You're about to hand-roll a layout that already exists here → use the pattern.
- The pattern doesn't quite fit → extend it in place, don't fork it.
- There's no pattern for what you need → build it in your product, and open a ticket to promote it once it stabilizes.
