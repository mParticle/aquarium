# Filter patterns

Two patterns are documented for filtering tables. **Reach for the Inline Toolbar by default.** Use Modal Filters only as an escape hatch when there are too many filter dimensions to fit inline.

## Inline Toolbar — recommended

A toolbar above the table holds: search on the right, 1–3 high-value filters on the left (Segmented for status, Select for environment, etc.), and an active-filter chip row that shows what is currently applied with x-to-remove.

**Why this is the default**

- Applied state is always visible. Users don't have to open anything to see what is filtered.
- One-click toggles for the most common filters (status, environment).
- No Apply-button ambiguity. Filters update the table as the user changes them.
- Composes naturally with sort (per-column header) and search (separate input).

See **Patterns / Inline Toolbar Filters**.

## Modal Filters — escape hatch

A "Sort & Filters" button opens a modal that collects sort direction plus many filter dimensions behind an Apply button. The modal is paired with visible filter chips beneath the toolbar so users can see what is currently applied without re-opening it.

**When to use**

- 5+ filter dimensions where an inline toolbar would be too crowded
- Power-user surfaces where filters change rarely once set
- Tables embedded in narrow layouts where a toolbar can't fit

**Tradeoffs to be aware of**

- Applied state is hidden inside the modal — chips below the toolbar are required, not optional.
- Apply-button semantics imply form validation; users hesitate, lose changes when they cancel.
- Combining sort + filter in one modal couples two distinct mental models; consider keeping sort on column headers if possible.

See **Patterns / Modal Filters**.

## Other filter primitives

- **Date range filters** — see [Date Range Filter](/?path=/docs/ux-patterns-data-exploration-date-range-filter--documentation) for the time-window pattern (presets + custom range).
- **Search** — `Input` with a search icon prefix and `allowClear`. Both filter patterns above include search as a separate input.
- **Per-column filters** — for filters tightly bound to one column (e.g. column-level multi-select), use Ant Design's column `filters` + `onFilter` props rather than a toolbar entry.
